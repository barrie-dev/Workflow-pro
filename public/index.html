Authenticatie

Interactieve schermen gebruiken een bearer token uit de login-response. Externe koppelingen gebruiken tenant API keys via X-API-Key.

POST /api/auth/login
Content-Type: application/json

{
  "email": "admin@demobouw.be",
  "password": "admin123",
  "mfaCode": "123456"
}
Authorization: Bearer <token>
X-API-Key: wfp_...

GET requests werken met read en modulescopes.
POST, PATCH en andere writes vereisen ook de scope "write".
POST /api/me/mfa/setup
POST /api/me/mfa/verify

{
  "code": "123456"
}

Na MFA-activatie geeft de verify-response recovery codes eenmalig terug. Die kunnen later als MFA-code gebruikt worden en worden server-side verbruikt.

Connectieregels

Gebruik API keys alleen voor server-to-server koppelingen. Adminacties zoals keybeheer, rapportgeneratie, backup restore en support consent vereisen een interactieve adminlogin.

Headers:
Authorization: Bearer <token>
X-API-Key: wfp_...
Content-Type: application/json
Scopes:
read - data lezen
write - data wijzigen
planning - planning endpoints
workorders - werkbon endpoints
billing - facturatie endpoints
integrations - koppelingen en sync
Rate limiting:
Publieke en API-routes worden begrensd per IP/route.
Elke API-key houdt usage count, lastUsedAt en lastEndpoint bij.

Integraties

Integraties gebruiken versleutelde secrets, field mapping en traceerbare sync logs.

Elke integratie geeft ook een syncSummary terug met laatste status, aantal successen, fouten, retries en aandachtstatus.

Een sync zonder geldige secret wordt als failed gelogd en zet de integratie op error, zodat pilots foutieve credentials meteen zien.

Field mappings worden server-side gevalideerd: local en remote veld zijn verplicht, richting is push, pull of both.

Een sync met ontbrekende of ongeldige field mapping wordt geblokkeerd en als failed gelogd.

Retry vereist een bestaande failed sync log en is idempotent: een reeds opgeloste failed sync krijgt geen dubbele retry-log.

GET /api/tenants/t_demo/integrations
POST /api/tenants/t_demo/integrations/connect
POST /api/tenants/t_demo/integrations/<id>/mapping
POST /api/tenants/t_demo/integrations/<id>/sync
POST /api/tenants/t_demo/integrations/<id>/retry

API keys

Tenant API keys worden gehashed opgeslagen. Externe systemen sturen de token mee via X-API-Key.

De admin ziet usage count, laatste gebruiktijd en laatste endpoint. Zo blijven toekomstige connecties traceerbaar zonder tokenwaarden opnieuw te tonen.

Nieuwe keys krijgen standaard een vervaldatum na 90 dagen. Een datum in het verleden wordt geweigerd.

Elke key moet minstens read of write hebben en minstens een concrete module-scope zoals planning, workorders, billing of integrations.

Rotatie kan scopes herstellen voor legacy keys. Zonder nieuwe vervaldatum krijgt de geroteerde key opnieuw 90 dagen geldigheid.

GET /api/tenants/t_demo/api-keys
POST /api/tenants/t_demo/api-keys
GET /api/tenants/t_demo/api-keys/governance?strict=true
POST /api/tenants/t_demo/api-keys/governance/run
POST /api/tenants/t_demo/api-keys/<keyId>/rotate
POST /api/tenants/t_demo/api-keys/<keyId>/revoke
{
  "label": "ERP test key",
  "scopes": ["read", "planning", "workorders", "integrations"],
  "expiresAt": "2026-12-31T23:00:00.000Z"
}
GET /api/modules/workorders?tenantId=t_demo
X-API-Key: wfp_...

POST requests vereisen ook de scope "write".
